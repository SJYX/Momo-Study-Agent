# 零凭证新用户上手改造计划

更新时间：2026-04-13
状态：Draft v1（待评审）
目标版本：v0.1 -> v0.2

## 1. 背景与目标

当前首次使用流程对“零凭证用户”（没有 MOMO Token、AI Key、Turso 配置）不友好，主要痛点：
- 向导存在强校验循环，无法跳过，用户可能卡死。
- 错误信息不分类型，无法快速定位是网络、权限还是配置问题。
- 凭证输入明文回显，存在泄露风险。
- 云端强制模式下的交互分支复杂，且存在“继续但已知失败”的路径。
- 文档缺少明确的新手最小路径与排错顺序。

本计划目标：
- 让零凭证用户在第一次运行时可以“可理解、可前进、可恢复”。
- 将首次成功率从“依赖用户经验”改为“依赖流程引导”。
- 保证安全基线（凭证不回显、不误存、不误传）。

## 2. 成功标准（验收口径）

### 2.1 体验指标
- 新用户首次运行时，3 分钟内能明确知道下一步做什么。
- 在缺失任意凭证时，流程不会死循环，提供至少一个可退出或稍后继续路径。
- 错误提示可定位到原因类别（网络/权限/格式/依赖）。

### 2.2 功能指标
- 向导支持三种模式：
  1) 立即配置并校验
  2) 先保存后校验
  3) 跳过并结束向导（保留“待配置”状态）
- 强制云端模式缺 Hub 配置时，不再提供“明知失败仍继续”的默认路径。
- 提供 preflight 检查入口，一次性输出缺失项和修复建议。

### 2.3 安全指标
- Token/API Key 输入默认隐藏回显。
- 日志中不出现完整凭证，仅允许掩码展示。
- 配置写入前有确认提示与风险提醒。

## 3. 范围定义

### 3.1 In Scope
- 新用户创建流程（profile 选择、向导、首次启动校验）
- 配置输入与校验反馈
- README 与环境模板中的新手指引
- preflight 检查脚本（命令行可调用）

### 3.2 Out of Scope
- AI 提示词质量优化
- 学习算法与迭代评分逻辑改造
- 云端同步性能大改（仅处理首次可用性）

## 4. 方案设计

### 4.1 交互重构（Wizard v2）
- 每个必填项加入“跳过/稍后配置”分支。
- 所有校验改为“最多重试 N 次 + 继续选项”，禁止无限循环。
- 失败提示分层：
  - 网络不可达
  - 凭证无效（401/403）
  - 依赖缺失（SDK/库未安装）
  - 服务端异常（5xx）
- 提供末尾总结：当前缺失项、可运行模式、下一步命令。

### 4.2 安全改造
- 使用隐藏输入读取敏感字段（MOMO Token、AI Key、TURSO_AUTH_TOKEN）。
- 输出和日志统一使用掩码（如前 4 + 后 2）。
- 保存 profile 前二次确认，明确“本地明文存储”风险提示。

### 4.3 强制云端模式兜底
- FORCE_CLOUD_MODE=True 且 Hub 缺失时：
  - 选项 A：立即配置 Hub
  - 选项 B：临时降级本次会话为本地模式
  - 选项 C：退出并打印修复清单
- 移除“继续但已知失败”的模糊路径。

### 4.4 Preflight 检查
新增命令：
- python tools/preflight_check.py --user <name>

输出内容：
- 用户 profile 存在性
- MOMO Token / AI Key / Provider 完整性
- Turso 用户库配置完整性
- Hub 配置状态
- FORCE_CLOUD_MODE 与当前配置是否冲突
- 一键修复建议（按优先顺序）

### 4.5 文档补全
- README 新增“零凭证新手路径（10 分钟）”
- .env.example 增加“最小可运行配置”与“云端可选配置”区分
- docs/dev 增加故障排查流程图（文本版即可）

## 7. 测试计划

### 7.1 手工测试矩阵
- Case A：完全零凭证

## 8. 风险与回滚

风险：
- 交互分支增多引入状态错误。
- 隐藏输入在部分 IDE 终端兼容性差。
- 强制模式策略变更可能影响既有自动化脚本。

缓解：
- 增加状态机单测。
- 对不支持隐藏输入的终端提供降级提示。
- 引入环境开关兼容旧流程（如 WIZARD_V2_ENABLED）。

回滚策略：
- 保留旧向导入口，出现线上问题可切回旧流程。
