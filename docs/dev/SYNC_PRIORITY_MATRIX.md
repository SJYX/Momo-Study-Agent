# SYNC_PRIORITY_MATRIX.md

## 目标

将“同步任务”从可能影响前端加载的后台活动，改造成有明确优先级、可抢占、可降级的调度体系。

核心原则：

1. 前端关键请求优先于任何同步
2. 今日任务优先于未来任务与补偿同步
3. 闲时尽量清空积压，高峰时自动让路

## 一、接口与查询分级清单

### P0（最高）前端关键链路

- `GET /api/session`
- `GET /api/users`
- `GET /api/stats/summary`（仅轻量统计版本）
- `GET /api/study/today`
- `GET /api/study/future`
- 静态资源与入口：`/`, `/assets/*`, `index.html`

要求：

- 不做全量未同步列表扫描
- 不做重查询拼接
- 不等待同步 worker 完成
- 目标：P95 稳定，超时优先返回降级数据

### P1 今日任务链路

- `POST /api/study/process`
- `POST /api/study/iterate`
- `GET /api/tasks/{task_id}`
- `GET /api/tasks/{task_id}/events`

要求：

- 允许写入必要业务数据
- 同步入队可做，但执行配额应高于未来任务、低于 P0 请求
- 任务进行中，低优先级补偿同步自动降速

### P2 未来任务链路

- `POST /api/study/process-future`
- 未来任务相关读取接口

要求：

- 可批量处理，但必须支持被 P0/P1 抢占
- 可延后同步，不影响结果展示

### P3 同步控制链路

- `GET /api/sync/status`
- `POST /api/sync/flush`
- `POST /api/sync/retry`

要求：

- 读操作优先返回计数和采样，不默认全量明细
- 手动触发同步可执行，但让路 P0/P1

### P4 自动补偿与 warmup

- profile warmup 异步未同步扫描
- 启动后补偿入队
- 定时补偿同步

要求：

- 仅在闲时执行
- 分批、可中断、可暂停
- 严禁占用 P0/P1 的资源预算

## 二、任务优先级矩阵（执行层）

同步项统一带元数据：

- `priority`: `P1|P2|P3|P4`
- `source`: `today|future|manual|warmup|retry`
- `created_at`
- `deadline_ms`（可选）

优先级顺序：

1. `P0 前端关键请求`（不入同步队列，直接抢占资源）
2. `P1 今日任务同步项`
3. `P2 未来任务同步项`
4. `P3 手动同步/冲突重试`
5. `P4 warmup/自动补偿`

并发预算建议（初始值）：

- 全局同步 worker 并发：`1~2`
- `P1`: 最大占比 `60%`
- `P2`: 最大占比 `25%`
- `P3+P4`: 最大占比 `15%`

当检测到 P0/P1 压力上升时：

- 立即暂停 `P4`
- 降低 `P3`
- 保留 `P1` 最小吞吐

## 三、闲时同步策略

闲时判定（任一不满足则非闲时）：

1. 最近 `N` 秒无 P0 请求高峰
2. 当前无进行中的 P1 今日任务
3. DB 锁等待低于阈值
4. API P95 未超阈值

闲时执行方式：

- 小批次拉取（如每批 `50~200`）
- 时间片执行（如每轮 `200~500ms`）
- 每轮后重新评估是否继续

## 四、数据库与查询约束

必须避免的模式：

1. 高频接口中 `get_unsynced_notes()` 全字段全量读取后 `len(...)`
2. 前端加载阶段触发冲突明细重查询
3. warmup 一次性全量扫描并立刻全入队

建议替代：

1. `COUNT(*)` 统计待同步数量
2. 冲突列表分页 + 默认小页
3. warmup 仅做“最小字段 + 分批入队”

## 五、SLO 与降级规则

建议监控阈值：

- `GET /api/stats/summary` P95 > 300ms
- `GET /api/study/today` P95 > 400ms
- DB 锁等待 P95 > 100ms

触发阈值后的自动动作：

1. 暂停 `P4` 自动补偿
2. `P3` 降到单批最小吞吐
3. `P1/P2` 仅保留必要同步动作
4. 关键接口返回降级字段（保证可用性）

## 六、验收标准

1. 页面首次进入与切换不因同步出现明显卡顿
2. 今日任务吞吐稳定，不被 future/warmup 挤占
3. 闲时队列可持续下降，非闲时能自动让路
4. 同步失败不会阻断前端关键链路
