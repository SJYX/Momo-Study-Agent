# SYNC_PRIORITY_MATRIX.md

## 目标

将“同步任务”从可能影响前端加载的后台活动，改造成有明确优先级、可抢占、可降级的调度体系。

核心原则：

1. 前端关键请求优先于任何同步
2. 今日任务优先于未来任务与补偿同步
3. 闲时尽量清空积压，高峰时自动让路

## 一、接口与查询分级清单

### P0（最高）前端关键链路

- `GET /api/session`
- `GET /api/users`
- `GET /api/stats/summary`（仅轻量统计版本）
- `GET /api/study/today`
- `GET /api/study/future`
- 静态资源与入口：`/`, `/assets/*`, `index.html`

要求：

- 不做全量未同步列表扫描
- 不做重查询拼接
- 不等待同步 worker 完成
- **支持前端预获取（Pre-fetching）**：接口必须保持极度轻量，以随时响应前端基于 `Hover` 的高频预加载请求，保障切页瞬间命中缓存。
- 目标：P95 稳定，超时优先返回降级数据

### P1 今日任务链路

- `POST /api/study/process`
- `POST /api/study/iterate`
- `GET /api/tasks/{task_id}`
- `GET /api/tasks/{task_id}/events`

要求：

- 允许写入必要业务数据
- 同步入队可做，但执行配额应高于未来任务、低于 P0 请求
- 任务进行中，低优先级补偿同步自动降速

### P2 未来任务链路

- `POST /api/study/process-future`
- 未来任务相关读取接口

要求：

- 可批量处理，但必须支持被 P0/P1 抢占
- 可延后同步，不影响结果展示

### P3 同步控制链路

- `GET /api/sync/status`
- `POST /api/sync/flush`
- `POST /api/sync/retry`

要求：

- 读操作优先返回计数和采样，不默认全量明细
- 手动触发同步可执行，但让路 P0/P1

### P4 自动补偿与 warmup

- profile warmup 异步未同步扫描
- 启动后补偿入队
- 定时补偿同步

要求：

- 仅在闲时执行
- 分批、可中断、可暂停
- 严禁占用 P0/P1 的资源预算

## 二、任务优先级矩阵（执行层）

同步项统一带元数据：

- `priority`: `P1|P2|P3|P4`
- `source`: `today|future|manual|warmup|retry`
- `profile_id`: **（关键）标识该同步任务所属的账户/词库 ID**
- `created_at`
- `deadline_ms`（可选）

优先级顺序：

1. `P0 前端关键请求`（不入同步队列，直接抢占资源）
2. `P1 今日任务同步项`
3. `P2 未来任务同步项`
4. `P3 手动同步/冲突重试`
5. `P4 warmup/自动补偿`

并发调度机制与预算（防写锁）：

鉴于底层 SQLite 存在并发写锁瓶颈，**严禁使用多线程并发写入**。
- 同步执行器（Worker）必须为**单线程串行执行（单消费者）**。
- 资源预算与优先级体现在**“带权轮询出队（Weighted Polling）”**策略上。

出队轮询概率/权重建议：

- `P1`: 提取概率 `60%`（或每轮最多拉取 6 个）
- `P2`: 提取概率 `25%`（或每轮最多拉取 2 个）
- `P3+P4`: 提取概率 `15%`（或每轮最多拉取 1 个）
- **防饿死（Starvation）保底**：连续处理 N 个 P1 后，如果有 P2 任务，强制插队放行 1 个 P2。

**多 Profile 协同调度控制：**

- **活跃状态赋权**：调度器实时感知当前处于 `Active`（用户正在界面交互）状态的 Profile，其产生的同步任务享有 `P1/P2` 的正常出队权重。
- **后台状态降级**：对于目前挂在后台的（非 Active）Profile，其所有自动同步请求强制被压制为 `P4`，仅在系统绝对闲时处理。
- **效果**：系统可同时挂载 N 个 Profile，但用户的当前前台操作**永远是全局最高优先级，绝不卡顿**。

中断与抢占粒度：

- **批次级抢占（Batch-level）**：禁止中断进行中的数据库事务以防回滚代价过大。高优请求到来时，等待当前小批次（如 20-50 条，< 100ms）执行完毕，下一轮优先拉取高优任务。

当检测到 P0/P1 压力上升时：

- 立即暂停 `P4`
- 降低 `P3`
- 保留 `P1` 最小吞吐

## 三、闲时同步策略

闲时判定（需满足所有条件）：

1. 最近 `N` 秒无 P0 请求高峰
2. 过去 10 秒内未收到 P1 级别请求，且 P1 队列为空
3. DB 锁等待低于阈值
4. API P95 未超阈值

状态切换防抖（Debounce）：

- **进入闲时**：必须**连续满足闲时条件超过 5 秒**，才真正启动 P4 等低优补偿任务，防止状态秒级跳动（Thrashing）。
- **退出闲时**：一旦任一条件不满足，**立刻响应**并暂停低优任务拉取。

闲时执行方式：

- 小批次拉取（如每批 `50~200`）
- 时间片执行（如每轮 `200~500ms`）
- 每轮后重新评估是否继续

## 四、数据库与查询约束

必须避免的模式：

1. 高频接口中 `get_unsynced_notes()` 全字段全量读取后 `len(...)`
2. 前端加载阶段触发冲突明细重查询
3. warmup 一次性全量扫描并立刻全入队

建议替代：

1. `COUNT(*)` 统计待同步数量
2. 冲突列表分页 + 默认小页
3. warmup 仅做“最小字段 + 分批入队”

## 五、SLO 与降级规则

建议监控阈值：

- `GET /api/stats/summary` P95 > 300ms
- `GET /api/study/today` P95 > 400ms
- DB 锁等待 P95 > 100ms

触发阈值后的自动动作：

1. 暂停 `P4` 自动补偿
2. `P3` 降到单批最小吞吐
3. `P1/P2` 仅保留必要同步动作
4. 关键接口执行**协同降级**：
   - 数据层面：返回降级字段（跳过耗时查询/统计，保证可用性）
   - 协议层面：API 响应中附加元数据（如 `meta: { _is_degraded: true, _sync_delayed: true }`）
   - UI 层面：前端感知降级状态并显示非侵入式提示（如：“系统繁忙，数据正在排队同步…”），避免用户误认为数据丢失。

## 六、验收标准

1. 页面首次进入与切换不因同步出现明显卡顿
2. 今日任务吞吐稳定，不被 future/warmup 挤占
3. 闲时队列可持续下降，非闲时能自动让路
4. 同步失败不会阻断前端关键链路
