# MoMo Web UI 执行计划（A+ 重构版）

更新时间：2026-05-02  
适用阶段：V2 规划与实现  
当前策略：抛弃历史包袱，效率优先，先重构交互主干再补细节

---

## 0. 一句话定位

将 MoMo Web 打造成 **以 Ops Monitor 为默认首页的高效执行控制台（Command Center）**，并新增独立只读 **Ops Monitor 监控台**：操作与监控双轨协同，默认最短路径完成任务闭环。

---

## 1. 产品目标与北极星指标

### 1.1 一级目标（3 个月）

1. 任务执行效率最大化（你已确认）
2. Today 批处理闭环可在单工作台完成
3. 出错后可快速定位并组级恢复
4. 保持 profile 隔离与数据安全边界不退化

### 1.2 北极星指标（KPI）

1. TTFS（Time To First Success）：进入 Today 后首次成功任务耗时
2. 批处理完成时长：从触发到终态（done/error）
3. 失败恢复时长：失败出现到完成重试的中位时间
4. 流程中断率：用户在中途离开/放弃的比例
5. 误操作率：触发高风险动作后撤销/报错比例

### 1.3 体验指标（UX）

1. 单次批处理关键点击数
2. 关键路径页面切换次数
3. 任务状态可解释性（是否无需看原始日志即可决策）

---

## 2. 设计原则（A+）

1. **效率优先**：主路径最短，默认信息平衡密度
2. **操作与监控分层**：Today Command Center 负责执行；Ops Monitor 负责全局态势
3. **一致语义**：Today/Future/Iteration 使用同一状态词典与操作模型
4. **平衡自动化**：默认可执行；高风险动作二次确认
5. **失败先分组**：失败按类型聚合，支持组级重试
6. **键盘友好**：提供 5-8 个高价值快捷键
7. **可观测优先**：所有关键状态可被结构化追踪
8. **开发期可用性优先**：Today 在开发期必须可用，不影响日常使用

## 2.1 开发期 Today 保护策略（已确认：平衡档）

1. 允许改 Today，但必须满足三条硬门禁：`开关保护 + 回归全绿 + 可回退`
2. 所有影响 Today 主路径的改动必须默认 `feature flag` 控制，支持快速关闭
3. 每次变更必须通过 Today 冒烟与关键回归后才可合入
4. 任何发布失败或体验明显退化时，优先执行开关回退而非在线修补
5. 章节推进顺序受该策略约束：优先做不破坏 Today 日常可用性的工作

## 3. 信息架构重构

## 3.1 顶层导航

1. Ops Monitor（默认首页，独立监控台）
2. Today Command Center（主执行入口）
3. Future
4. Iteration
5. Word Library
6. Sync Status
7. Preflight
8. Settings

## 3.2 页面职责边界

1. Today Command Center：筛选、执行、进度、失败处理、重试、结果摘要
2. Ops Monitor：全局队列健康、任务热区、失败趋势、系统状态（只读）
3. Future/Iteration：沿用 Today 语义，保留场景特化字段
4. TaskDrawer：从“日志抽屉”升级为“执行上下文抽屉”（结构化信息为主，日志折叠）

## 4. 核心交互模型（与你确认的一致）

## 4.1 主工作流（Today）

1. 进入页面后自动恢复上次筛选与排序
2. 默认排序：价值优先（可切换）
3. 选择批次并执行（平衡自动化）
4. 实时查看总进度 + 行级状态
5. 失败自动分组
6. 组级重试或跳过
7. 产出结果摘要并可一键回看明细

## 4.2 失败分组模型

失败分组至少包含：

1. AI 调用失败
2. 数据冲突（sync_conflict）
3. 同步失败（sync_failed）
4. 参数/数据不合法
5. 系统异常

每个分组支持：

1. 查看组内条目
2. 查看共性原因
3. 组级重试（带二次确认策略）
4. 局部排除后再重试

## 4.3 自动化与确认策略（平衡）

1. 普通执行：一次触发直接运行
2. 高风险动作二次确认（批量覆盖、强制重跑、可能影响持久化的一次性操作）
3. 确认弹窗必须展示影响范围与可撤回性说明

## 4.4 快捷键（中等强度）

预留 5-8 个：

1. 打开筛选 `F`
2. 执行当前批次 `Ctrl+Enter`
3. 失败分组面板 `G`
4. 组级重试 `R`
5. 全局搜索 `Ctrl+K`
6. 切换排序 `S`
7. 打开任务抽屉 `D`
8. 刷新状态 `Shift+R`

## 5. 独立 Ops Monitor（A+ 增量核心）

## 5.1 定位

只读、全局、态势型监控页，不承载复杂操作，避免干扰执行主路径。

## 5.2 监控内容

1. 当前运行任务总览（按 profile/页面来源/状态）
2. 失败热区（按失败类型、模块、时间窗口）
3. 队列深度与延迟
4. 最近 N 次任务成功率曲线
5. 系统健康信号（API、DB、Sync、SSE）

## 5.3 监控台交互

1. 支持 drill-down 到具体任务
2. 一键跳转到对应 Today/Future/Iteration 上下文
3. 支持时间范围切换（15m/1h/24h）
4. 提供只读导出（便于复盘）

---

## 6. 数据契约与状态机

## 6.1 统一状态语义

任务状态：`pending/running/done/error/canceled`  
行级状态：`pending/running/done/error/skipped/sync_pending/sync_conflict/sync_failed`

## 6.2 关键契约要求

1. 前后端统一 `{ok,data,error}`
2. SSE 事件保持结构化，日志仅作附加信息
3. 失败事件必须附带可分组字段（error_type/error_code/phase）
4. 所有重试动作必须具备幂等与可追踪 request_id

## 6.3 状态一致性规则

1. 总进度与行级聚合结果必须一致
2. profile 切换必须清理旧订阅与旧上下文状态
3. 任务终态后禁止继续推送运行态事件

---

## 7. 安全与风险控制

1. 保持本地绑定策略（默认 127.0.0.1）
2. 延续 profile 隔离、同 profile 重任务互斥
3. 高风险动作二次确认
4. 避免误触：危险按钮与常用按钮视觉隔离
5. 保留 auth/tenant 扩展位，不在本期硬做认证体系
