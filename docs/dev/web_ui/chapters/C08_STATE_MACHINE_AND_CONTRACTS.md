# C08 状态机与数据契约

状态：confirmed  
更新时间：2026-05-02

## 1. 章节目标

统一任务状态流转、错误语义与幂等协议，确保前后端行为一致、可追踪、可恢复。

## 2. 已确认决策

1. 任务终态不可逆：终态后不允许回流到 running
2. 错误码粒度：领域级错误码（task/sync/ai/profile...）
3. 重试幂等键：前端生成 request_id 并透传
4. SSE 断线重连上限：3 次，超过后提示人工介入
5. 错误码格式：稳定编号制（如 `E_TASK_001`）
6. SSE 重连超限后：自动降级到手动刷新模式
7. request_id 去重窗口：30 分钟
8. 错误建议动作：统一展示（retry/check network/open monitor）

## 3. 章节计划（可执行）

1. 状态机层
   - 固化终态不可逆规则
   - 校验非法状态回流并拦截
2. 错误契约层
   - 建立领域级错误码字典
   - 引入稳定编号规范
3. 幂等层
   - 前端生成并透传 request_id
   - 后端按 30 分钟窗口做去重
4. SSE 稳定性层
   - 断线最多重连 3 次
   - 超限后自动降级手动刷新
5. UX 层
   - 统一错误建议动作组件

## 4. 验收标准

1. 终态任务不会回流到 running。
2. 错误码遵循领域+稳定编号规范。
3. request_id 前端生成且后端 30 分钟去重生效。
4. SSE 重连超过 3 次后自动降级手动刷新。
5. 各页错误提示统一提供建议动作。

## 5. 风险与回退

1. 风险：错误码升级影响历史兼容。
   - 处置：维护兼容映射表，逐步迁移。
2. 风险：30 分钟去重误伤真实重试。
   - 处置：手动刷新触发新 request_id 并显式提示。
3. 风险：SSE 降级后实时性下降。
   - 处置：在 UI 明示已降级并提供手动刷新入口。
