记录所有"为什么不那样做"的决策，防止未来重蹈覆辙。每次大功能迭代后更新。

## DEC-001 · 不干预主线复习队列

**日期**: 2026-04-10
**场景**: 薄弱词重炼后需要强化复习
**否定方案**: 调用 `POST /api/v1/study/advance_study` 将薄弱词强推进今日复习
**采用方案**: 将薄弱词汇聚到私有云词本 `MomoAgent: 薄弱词攻坚`（UNPUBLISHED）
**原因**: 用户明确表示不希望脚本影响其既定的每日打卡节奏。云词本作为独立模块，用户可自主选择时机消灭顽固词汇。

## DEC-002 · 数据库双轨策略

**日期**: 2026-04
**场景**: 生产数据库选型
**否定方案**: 纯云端 Turso / 纯本地 SQLite
**采用方案**: 本地 SQLite（`data/history_*.db`）+ Turso 云端双向同步
**原因**:
- 网络抖动时本地继续工作，不阻塞 AI 生成流程
- 程序启动时本地数据库结构升级不依赖云端连接
- 多设备通过云端同步，本地作为热数据缓存

---

## DEC-003 · 禁止在云端连接上设置 row_factory

**日期**: 2026-04-10
**场景**: SQLite 查询结果映射为字典
**否定方案**: `conn.row_factory = sqlite3.Row`
**采用方案**: `_row_to_dict(cursor, row)` 兼容函数
**原因**: Turso 的 `libsql.Connection` 对象（`builtins.Connection`）不支持 `row_factory` 属性赋值，直接抛出 `AttributeError`，导致选项 3 意外崩溃。

---

## DEC-004 · Token 验证使用低门槛探针

**日期**: 2026-04-10
**场景**: 新用户配置向导中验证 MaiMemo Token 合法性
**否定方案**: `api.get_today_items(limit=1)` 作为验证探针
**采用方案**: `api.get_study_progress()` 作为验证探针
**原因**: 新用户若尚未在 App 内选定词书，`get_today_items` 会返回异常结构或空数据，导致系统误判 Token 无效，卡死初始配置流程。

## DEC-007 · Turso 新用户数据库创建

**日期**: 2026-04-10
**场景**: 新用户 Profile 初始化时需自动准备专属数据库
**否定方案**: 让用户手动在 `data/profiles/{user}.env` 中配置 `TURSO_DB_URL`/`TURSO_AUTH_TOKEN`
**采用方案**: 在 `core/config_wizard.py` 新增可选 Turso 数据库创建步骤，调用 Turso REST API:
`POST /v1/organizations/{organizationSlug}/databases`
**原因**:
- 降低新用户配置门槛，避免因环境变量配置错误导致同步失败
- 保证每个用户独立数据库名 `history_{username}`，坚持用户隔离原则
- 便于后续自动化部署、新用户创建和云端同步体验一致性

## DEC-005 · 批量同步代替逐条请求

**日期**: 2026-04
**场景**: 本地与云端数据库同步
**否定方案**: 逐条记录发起网络请求
**采用方案**: 以可配置批量方式复用网络请求，当前默认批次由运行配置控制
**原因**: 逐条请求在数据量大时产生"假死"感（数千次网络往返），批量化后显著降低延迟，消除用户等待压力。

---

## DEC-006 · Prompt 文件统一归入 docs/prompts/

**日期**: 2026-04-10
**场景**: Prompt 文件管理
**否定方案**: Prompt 文件散落在根目录
**采用方案**: 统一归入 `docs/prompts/`，路径通过 `config.py` 中的常量引用
**原因**: 随项目增长，根目录文件夹混乱，影响导航效率。Prompt 文件属于文档资产，归入 docs/ 体系更合理。

---

## DEC-008 · 中央用户信息库 + 管理员权限控制

**日期**: 2026-04-11
**场景**: 多用户管理、创建云端数据库权限管理、用户行为追踪
**否定方案**:
- 用户信息分散存储在各自的个人库中
- 创建云端数据库无权限限制，任何用户都可以创建
- 不记录用户访问日志和会话信息
**采用方案**:
- 创建全局中央库 `momo_users_hub`（6 张表）记录所有用户元数据
- 创建云端数据库时需验证**管理员密码** (`ADMIN_PASSWORD_HASH`)
- 记录用户会话、IP、客户端信息和操作日志
- 所有时间戳使用 ISO 8601 含时区格式
- 敏感数据（API Key）使用 Fernet 对称加密
- **中央 Hub 双轨策略**：优先尝试连接 Turso 云端 Hub，无配置或网络不通时自动回退至本地 SQLite (`data/momo-users-hub.db`)
- **命名约束**：Turso 数据库名不支持下划线，中央库统一使用 `momo-users-hub`。
- **Token 自提升机制**：使用一次性母令牌生成名为 `momo-hub-manager` 的 Org-scoped 平台令牌并存入 `.env`，避免高权令牌长期暴露在日志或配置文件中。
**原因**:
- 集中管理所有用户配置，便于统计、审计和故障排查
- 管理员密码保护防止误操作和权限滥用
- 用户会话追踪提供安全审计线索
- 时区感知和加密存储符合企业级数据安全要求
- **本地回退**确保了即使在没有云端凭据的离线/开发环境下，多用户逻辑和管理员权限验证依然能闭环工作
- **自提升令牌**实现了权限隔离，Hub 管理器仅拥有必要的资源管理权限，符合安全最小化原则

## 待后续考量 · 同一 voc_id 的多版本笔记保留策略

**场景**: 当 prompt 版本或 AI 引擎发生变化时，是否需要为同一个 `voc_id` 保留多份历史笔记用于对比和回溯
**当前处理**: 暂不改 `ai_word_notes` 主表结构，仍按 `voc_id` 覆盖为当前可用的主结果
**原因**:
- 当前目标是优先优化今日任务/未来计划的热路径，不希望引入版本化存储带来的额外查询和同步复杂度
- 主表保持单条记录更适合当前查重、同步和回写流程，避免同一词条在主结果层面出现多份并存
- 如果后续要做 prompt 对比、A/B 评估或回溯分析，再单独评估版本化存储或历史表方案
