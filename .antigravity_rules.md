# Momo Study Agent - AI Coding Rules & Patterns

本文件记录了本项目在开发过程中沉淀的核心逻辑模式和工程规范。AI 在后续修改和扩展代码时，必须严格遵守以下准则。

## 1. AI 鲁棒性与 JSON 提取 (Robust JSON Parsing)
- **准则**：Gemini 生成长 JSON 数组时，末尾极易产生非 JSON 的幻觉乱码（如 `]道路]`）。
- **实现模式**：
    - **禁止**直接使用 `json.loads(text)`。
    - **必须**先通过 `_extract_json_array(text)` 使用括号计数器找到第一个合法闭合的 `]` 位置。
    - **必须**使用 `rfind('```json')` 或类似的 Markdown 清洗逻辑。

## 2. 数据安全与写入限制 (Data Safety & Writing Restriction)
- **核心原则**：未经过测试脚本验证及用户明确许可，**严禁**向墨墨 API 写入真实数据（POST/PUT 操作）。
- **准则**：
    - 在任何涉及到写入墨墨的脚本运行之前，必须先使用测试脚本在 `test.db` 环境下完整运行一遍。
    - 所有的主流程脚本（如 `main.py`）默认必须设置 `DRY_RUN = True`。
    - AI 在修改代码时，不得在未经询问的情况下将 `DRY_RUN` 设为 `False`。
    - 所有测试脚本必须指向 `config.TEST_DB_PATH`。
    - `db_manager.py` 必须支持通过参数传递 DB 路径，而不是硬编码。
    - 严禁在测试脚本中调用会向 Maimemo API 发送 POST/PUT 请求的方法（除非显式开启 `DRY_RUN=False`）。

## 3. 批量处理策略 (Batching Strategy)
- **准则**：为平衡 API 成本与 token 限制，推荐的批次大小为 **20 个单词**。
- **考量**：
    - 当批次 > 25 时，AI 输出格式崩溃的概率显著增加。
    - 批次处理时必须包含 `ielts_focus` 和 `memory_aid` 等关键维度，且需监控单词分析深度。

## 4. 终端兼容性 (Windows Compatibility)
- **准则**：在 Windows 环境下，输出 Unicode（如 Emoji）常导致 `UnicodeEncodeError`。
- **实现模式**：
    - 在脚本入口处，推荐使用 `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")`。
    - 日志输出中应避免使用非常规 Unicode 字符，或提供 ASCII 备选。

## 5. 配置中心化 (Centralized Config)
- **准则**：严禁在各个模块中分散读取 `.env`。
- **实现模式**：
    - 统一引用 `from config import ...`。
    - `config.py` 是唯一的环境变量入口和默认值定义处。

## 6. 日志规范 (Minimalist Logging)
- **准则**：采用“内容导向”的极简日志风格。
- **风格示例**：
    ```text
    🚀 [Batch 1] 开始请求 10 个单词...
    ⏳ 休息 20 秒以保护 API...
    ✅ 处理完成：biography (voc_id=...)
    ```
